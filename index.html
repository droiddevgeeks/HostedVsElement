Hosted Checkout vs Element

Three ways to take a payment. One question sorts them: how much of the payment experience do you want to build and own?

The one-paragraph version

With Hosted Checkout, Cashfree owns the payment screen. The moment your user taps Pay, we render the UI, list every payment mode — card, UPI, net banking, wallets — handle the bank redirects and UPI app hand-offs, and hand you back a result. You build almost nothing. With Element, you build the payment UI — your own card fields, your own UPI app list, your own design language — and process the payment through a Cashfree SDK (the Mobile SDK on Android/iOS, or Atom JS on web), which handles the plumbing underneath. And with Direct API, you build the UI and orchestrate the whole payment yourself against our APIs — no Cashfree SDK in the loop. The rule of thumb: the further right you move, the more control you get and the more you own.

The quick way to tell them apart: Hosted = our UI, our processing. Element = your UI, and a Cashfree SDK (the Mobile SDK, or Atom JS on web) does the processing. Direct API = your UI, and you orchestrate the processing yourself against our APIs, with no Cashfree SDK in the loop. The only real difference is how much of that chain you hand to us.

Three things stay identical wherever you land:

  1. You always create an order first. Your backend calls Cashfree's Create Order API and gets back a payment_session_id. Nothing happens client-side without it.
  2. You always verify the final status server-side. The client result is a hint, never the source of truth.
  3. Your secret keys never touch the client. Order creation and status checks live on your backend.

What changes is everything between those two endpoints.

02 — Option one

Hosted Checkout

UI: Cashfree  ·  Redirects / UPI / OTP: Cashfree  ·  Your job: create the order, open checkout, read the result

When the user taps Pay Now / Proceed to Pay, Cashfree takes over the screen. We render the full payment-mode list, capture the instrument, run 3DS/OTP for cards, fire the UPI collect/intent, and push the user to the bank page for net banking — all of it. You get control back only when the payment reaches a terminal state.

hosted-checkout — flow
App/Web  ──(1) initiate──▶  Merchant Backend  ──(2) Create Order──▶  Cashfree
                                            ◀─ payment_session_id ─
App/Web  ◀──(3) session_id─────────────
   │
   │ (4) open Cashfree checkout with payment_session_id
   ▼
Cashfree Checkout UI  ── renders modes, captures instrument,
                       handles 3DS/OTP, UPI, bank redirect
   │
   ├─(5a) result ──▶ App/Web        (UI hint)
   └─(5b) webhook ──▶ Merchant Backend
                          │
                          └─(6) Get Order / Get Payment ──▶ Cashfree  (source of truth)

Watch it: tap Pay, and Cashfree's checkout takes over until the result comes back.

On each platform

  • Web: load the Cashfree JS SDK and call cashfree.checkout({ paymentSessionId }) — as a redirect or a drop-in modal.
  • Mobile (Android/iOS): the Cashfree Mobile SDK exposes a checkout contract — hand it the payment_session_id, the SDK opens the hosted flow and returns via callback.

Why pick it

  • Fastest time-to-live. Days, not weeks — one screen open, one result callback.
  • Lowest PCI scope. Raw card data never enters your app or server, so your compliance burden stays minimal.
  • Free upgrades. New modes, banks, UPI and conversion improvements land automatically. You ship nothing.

What you give up

  • UI control. It's Cashfree's screen — themeable, but not yours. There's a context switch from your app's look-and-feel.
  • Flow control. You can't reorder steps or inject your own logic mid-payment.
03 — Option two

Element

Element means your UI, our SDK. You build the payment screen — your card fields, your UPI app grid, your net-banking list, your wallet tiles — and process the payment through a Cashfree SDK: the Mobile SDK on Android/iOS, or Atom JS (with Element components) on web. Cashfree never renders a screen; the SDK does the heavy lifting underneath.

You collect the instrument in your own components, then pass it to the SDK through a typed contract (a card-payment object, a UPI-payment object). The SDK calls the Cashfree backend, handles the bank-page redirect / UPI app launch / OTP screen, reconciles, and fires a callback. You then confirm status server-side.

element-sdk — flow
Your UI (card fields / UPI list) ── collects instrument
   │
   │ pass instrument via SDK contract (e.g. CFCardPayment / CFUPIPayment)
   ▼
Cashfree SDK ──▶ Cashfree Backend
   │  (SDK handles bank redirect, UPI app launch, OTP)
   ▼
SDK callback: onPaymentVerify / onPaymentFailure ──▶ Your app
   │
   └─ Your backend ── Get Payment status ──▶ Cashfree  (verify) + webhook

Watch it: your custom UI collects the instrument, then the SDK handles the redirect and reconciliation.

  • Mobile: one Cashfree Mobile SDK serves both Hosted and Element. The contract you call decides the flow — pass a session for hosted, or pass a specific instrument for element. Same dependency, different entry points.
  • Mobile — custom card component (for non-PCI merchants): the SDK also ships a Cashfree-provided card-number input you embed directly inside your own screen. It looks like part of your UI, but Cashfree captures the card number and processes the payment — raw PAN never touches your code. You get the Element look-and-feel without taking on PCI scope. This is the sweet spot for merchants who want a custom card flow but aren't PCI-certified.
  • Web: Cashfree Element components (and Atom JS) give you composable building blocks — a secure card component you mount into your own page — while keeping sensitive-field handling inside Cashfree-controlled context. Same idea as the mobile custom card component: your layout, our secure field.

The big win: you own the look and feel, but you don't hand-roll the messy parts — redirect handling, UPI intents, OTP capture, retries, reconciliation. The SDK is also a PCI shield: when card capture flows through Cashfree's secure fields rather than your own raw inputs, your compliance scope stays far lower than going Direct API.

Rule of thumb

Element is the default path when you want your own UI. Only drop to Direct API (below) when you've hit a concrete wall the SDK can't clear.

04 — Option three

Direct API

The UI is still yours — but unlike Element, there's no Cashfree client SDK in the loop. Your app talks to your backend, your backend talks to Cashfree's Pay Order API, and Cashfree responds with an action: a redirect URL, a UPI intent, an OTP step. You open the bank page, you launch the UPI app, you submit the OTP, you track and verify status. Cashfree is purely backend APIs here.

direct-api — flow
Your UI ── collects instrument
   │
App ──▶ Merchant Backend ──(Pay Order)──▶ Cashfree
                          ◀─ action: redirect URL / UPI intent / OTP step ─
   │
App ── YOU open bank page / launch UPI app / submit OTP
   │
App/Backend ── poll Get Payment + webhook ──▶ Cashfree  (verify)

Watch it: your backend calls Pay Order, then your code handles the redirect and status — no SDK.

Pick this only if you need something the SDK won't give you: a fully custom redirect/app-launch experience, server-to-server flows (headless, IVR, your own native bridge), or you're consolidating multiple PSPs behind one in-house payment layer. The cost: you now own redirect handling, UPI intent edge cases, OTP UX, retry logic — and the heaviest PCI burden, since raw card data hitting your servers widens your compliance scope.

05 — Platform view

Web vs Mobile: what you actually import

The spectrum is the concept; here's how it maps to real dependencies.

Web

You want…Use
Hosted CheckoutCashfree JS SDK — cashfree.checkout({ paymentSessionId }), as redirect or drop-in modal
Element (your UI)Cashfree Element components mounted into your page, with Atom JS providing the payment contract for instrument submission
Direct APIYour backend orchestrates Cashfree's Pay Order API; you handle redirects, UPI, OTP, and status yourself

Key web nuance: even in Element, sensitive card capture should run through Cashfree-controlled secure components/iframes so your page DOM never sees the raw PAN. That's what keeps a browser-based Element integration from widening into full PCI scope.

Mobile (Android / iOS)

The important fact: there is one Cashfree Mobile SDK, and it handles both Hosted Checkout and Element. You don't pick a different library — you pick a different contract:

  • Pass a payment_session_id to the checkout entry point → Hosted flow (Cashfree's UI opens).
  • Pass a specific instrument object (card / UPI / net banking) → Element flow (your UI already collected it; the SDK just processes).

In both cases the SDK owns the hard parts on-device: launching the UPI app via intent, handling the bank page in a secure web view, OTP capture, and firing a verification callback at the terminal state. Direct API is available too — but on mobile you'd be re-implementing UPI intents and redirect web views by hand, which is exactly what the SDK exists to spare you.

07 — Don't skip this

Status verification: the part everyone gets wrong

This gets its own section because it's the #1 integration bug, and it behaves differently across the spectrum.

The trap

The client result is never the source of truth. A user can close the app mid-redirect, lose network right after paying, or hit back on the bank page. The on-device/in-browser callback can say "success," "failure," or nothing at all — none of it authoritative.

The authoritative answer always comes from two server-side mechanisms, and you should use both:

  1. Webhooks — Cashfree pushes the terminal payment status to your backend. Your primary, push-based signal.
  2. Get Order / Get Payment status API — your backend pulls the status on demand (when the user lands back, or as a reconciliation sweep). Your fallback for missed webhooks.

How responsibility shifts across the spectrum:

  • Hosted Checkout: Cashfree reconciles internally and hands you a result; you still confirm via webhook + status API before fulfilling.
  • Element: the SDK fires a verification callback after its own reconciliation, but you still confirm server-side before granting value.
  • Direct API: there's no SDK reconciliation. You are fully responsible for polling and webhooks. Skipping it here doesn't degrade gracefully — it breaks.

The one rule

Never fulfill an order — ship goods, unlock a subscription, mark paid — off a client callback alone. Wait for the server-confirmed terminal state.

09 — Decide

Pick X if…

Hosted Checkout

You want to be live this week, you don't have a dedicated payments team, payment UX isn't a competitive differentiator, or you want PCI scope as small as it gets.

→ Default for most merchants

Element

Payment UX is part of your brand — card fields and a UPI grid that look native, no jarring context switch — but you have no appetite for hand-rolling redirects, UPI intents, OTP capture, and reconciliation. Custom look, none of the dangerous plumbing.

→ Default for design-led, high-volume merchants

Direct API

You have a real payments engineering team and a concrete requirement the SDK can't meet: a headless/server-to-server flow, your own multi-PSP abstraction, or a redirect/app-launch experience you must control end to end. Go in knowing you own the full maintenance and PCI burden.

→ Only when you've hit a real wall

TL;DR

  • Hosted = our UI, our plumbing. Create an order, open checkout, confirm status. Least work, least control.
  • Element = your UI, our SDK. You build the screen; the Cashfree Mobile SDK (or Atom JS on web) drives network + redirects + reconciliation.
  • Direct API = your UI, and you orchestrate the whole payment yourself against our APIs — no Cashfree SDK in the loop. Most control, most responsibility.
  • One Mobile SDK covers both Hosted and Element; on web it's the JS SDK for Hosted, Element components + Atom JS for Element.
  • Whatever you pick: create the order server-side, and confirm the final status server-side before you fulfill.
